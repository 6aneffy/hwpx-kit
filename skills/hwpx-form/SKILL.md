# hwpx-kit: HWPX 양식 자동화

CLI `hwpx-kit`으로 hwpx 문서를 다룬다. 항상 `--json`을 붙여 실행하고 JSON 봉투(`ok`/`data`/`warnings`/`error`)를 파싱한다. 종료 코드: 0 성공, 1 오류, 2 부분 성공(unmatched 있음).

## CLI가 없다면 (hwpx-kit: command not found)

임기응변도 성급한 설치도 말고 이 순서 그대로:
1. **먼저 `python -m hwpx_kit.cli --version`** — 이미 설치됐는데 PATH에만
   안 잡힌 경우가 제일 흔하다. 되면 설치 불필요, 이후 모든 호출을
   `python -m hwpx_kit.cli ...`로 하면 끝
2. 그것도 없으면 Python 확인 (`python --version`, 3.10+ 필수) — Python이
   없으면 사용자에게 설치 안내: `winget install Python.Python.3.12` 또는
   python.org ("Add to PATH" 체크), 설치 후 새 터미널
3. 마지막으로 설치 — 사용자에게 시키지 말고 네가 직접:
   `pip install --user git+https://github.com/6aneffy/hwpx-kit.git`
   → `python -m hwpx_kit.cli --version` 확인 후 원래 작업 계속

기존 문서(계획안·보고서 작성본)를 양식 삼아 **다른 사업 문서를 새로 쓰는** 요청도
같은 흐름이다 — 내용(사업명·배경·커리큘럼 등)은 네가 초안을 짜서 사용자와 확정하고,
문서화는 read → text:/table: 교체로 한다. 서식은 원본 그대로 유지된다.

## 신작·전면 교체 요청은 정보 게이트부터 (바로 채우지 말 것)

"이 기획서를 ○○사업으로 바꿔줘" 류 요청 처리 순서:

0. **양식 파일 위치 확인** — 파일이 대화에 안 보이면 "양식 파일이 어디 있나요?
   경로를 알려주시거나 끌어다 놓아 주세요"부터. 특정 폴더 구조(incoming/ 등)를
   가정하지 말 것. 양식이 아예 없다면 doc-create 스킬의 백지 경로로
1. **필수 정보 체크** — **문서 유형에 맞는 필수 항목**이 다 모일 때까지
   물어보고, 최소 충족 후 작업 시작. 부족한 항목만 콕 집어 물을 것:
   - 계획서·기획서(행사/교육/사업): 제목 / 주최·기관 / 일시·기간 / 장소 /
     대상·정원 / 핵심 내용(과정 구성·회차 등)
   - 신청서: 신청인·소속·연락처 / 신청 대상·사유 / 기간
   - 보도자료: 제목 / 배포·보도 시점 / 담당 부서·연락처 / 핵심 내용
   - 그 외 유형: 그 문서를 읽고 빈 곳이 요구하는 정보를 스스로 도출해서 질문
   선택 확인: 예산 포함 여부, 명단 실명 표기 등 유형별 부속.
   사용자가 안 준 서술부(추진배경·목적 등)는 Claude가 초안 — 이건 안 물어도 된다.
   **질문은 AskUserQuestion 도구(선택지 UI)로** — 항목당 합리적 후보 2~4개 +
   추천 표시를 제시하면 사용자가 클릭으로 답한다 (자유 입력은 Other로 가능).
   한 번에 최대 4개 항목까지 묶어서. 답이 이상하거나 잘려 보이면 재확인
2. **구조 대조 보고**: read/analyze로 원본 구조 vs 새 내용을 대조해
   안 맞는 표·섹션 목록(예: 원본 20일 커리큘럼 표 vs 새 10일 과정)을 보여주고
   진행 방식을 사용자와 확정:
   - (a) 사용자가 한글에서 구조 정리(불필요 표·행 삭제) 후 파일 다시 제공 —
     가장 깔끔. 행 높이가 어긋나면 `row-height`로 도구가 정돈
   - (b) **기본값: 안 맞는 내용은 비운다** — 표는
     `hwpx-kit table-clear <파일> --table N [--rows A-B] --out <결과>`,
     문단은 `delete:<원문>` 키. 원본 내용을 그대로 남겨두면 안 됨 (엉뚱한
     교수명·일정이 새 문서에 남는 사고 — 실증됨). 빈 칸으로 남겨야 사용자가
     보고 "이 표는 지워달라 / 여긴 이런 내용 채워달라"를 판단할 수 있다.
     비운 자리에 새 항목(예산 항목명 등)을 앉힐 땐 라벨이 없으니
     `hwpx-kit table-set <파일> --table N --set "R,C=값" [--set ...] --out <결과>`
     (좌표는 analyze/table-map의 0-기준, 병합 셀은 anchor 좌표로).
     **셀 6개 이상 대량 기입은 `--data <셀.json>` 파일로** ({"R,C": "값"} 형식)
     — --set 수십 개를 한 명령에 넣으면 셸 명령 길이 한계로 깨진다 (실증됨).
     표 셀 상태 확인은 `hwpx-kit table-map <파일> [--table N] --json` —
     원시 XML을 직접 파싱하지 말 것
   - 원본 내용 잔존은 사용자가 명시적으로 요청할 때만
3. 내용 초안 확정 → 교체 → validate → 한글 육안 확인 안내 (기존 흐름)

## 양식 채우기 (주 사용 흐름)

1. **분석**: `hwpx-kit analyze <양식.hwpx> --json`
   - `data.fields[]`의 각 항목에 `fill_key`가 있다 (`clickhere:이름`, `marker:키`, `table:라벨`)
   - `ambiguous: true`인 표 라벨은 같은 라벨이 여러 곳 — fill_key에 문서 순서
     출현 번호가 붙어 나온다 (`table:담당 부서#1`, `table:담당 부서#2`).
     그 키 그대로 fill에 쓰면 각각 구분해 채워진다 (보도자료의 담당부서 2개 블록,
     보도시점 온라인/지면 행 등). `current` 값으로 어느 칸인지 판단하고, 애매하면
     사용자에게 확인
   - `prefilled: true`는 템플릿 기본값(`current`)이 이미 든 칸 — fill 하면 덮어써진다. 공문의 수신/제목처럼 바꿔야 하는 칸이 보통 이것. 값 셀이 다음 칸의 라벨로 잡히는 잡음 후보(전화번호→이메일 연쇄 등)는 무시
   - 라벨에서 오른쪽이 아닌 칸은 방향 체인으로 지정: `table:라벨 > down`,
     `table:과 장#1 > right > right` (이름 다음 전화 칸). 방향은 left/right/up/down
   - **값은 원본 칸 구성대로 쪼개서** — 이름 칸(`current`가 `○○○`)엔 이름만,
     전화/이메일은 그 오른쪽 칸에 체인으로 따로. 좁은 칸에 긴 값을 합쳐 넣으면
     줄바꿈으로 행 높이가 커져 표 모양이 무너진다
   - **작성 완료된 문서**(계획안·보고서 작성본)를 양식 삼으면 필드가 수백 개
     나온다 — 내용 셀이 다음 셀의 라벨로 잡히는 잡음이 대부분이다.
     `prefilled: false`(빈칸)와 사용자가 바꾸겠다고 한 항목만 추리고 나머지는 무시.
     빈 표(예산표 등)는 라벨 평키 + `> right` 체인 + `#N`으로 행 단위 채우기
   - **표 구조 변경(행·열 추가/삭제, 병합)은 불가** — 셀 내용만 바꿀 수 있다.
     행을 "비우기"는 내용만 지우는 것이라 **빈 행 껍데기가 표에 남는다.**
     새 내용이 표 크기와 안 맞으면(커리큘럼 20일→15일 등) 채우기 전에 멈추고
     사용자에게 요청: "한글에서 이 표를 N행으로 맞춰(행 추가/삭제) 다시 주시면
     이어서 채우겠다." 구조가 맞은 파일을 받아 진행하는 게 정답 순서 —
     빈 행을 남기거나 내용을 우겨넣지 말 것
   - 사용자가 행을 추가하면 **높이가 기본값으로 붙어 들쭉날쭉할 수 있다** —
     그건 사용자에게 시키지 말고 도구로 정돈:
     `hwpx-kit row-height <파일> --table <표인덱스> --like <기준행> --rows <대상행들> --out <결과>`
     (인덱스는 analyze의 table_index/row 그대로 0-기준, 기준 행은 병합 없는 행으로)
   - **장(章)이나 표를 늘려야 할 때** (템플릿은 5장인데 8장 필요 등): 장 제목
     박스는 실은 1x3 표다 —
     `hwpx-kit table-copy <파일> --table <헤더표 인덱스> --after-text "<삽입 위치 문단 원문>" --out <결과>`
     로 서식째 복제한 뒤 `table-set`으로 번호·제목 기입. 내용 표(커리큘럼 등)도
     같은 방식. **앵커 매칭 규칙 (소스 뒤질 필요 없음)**: 문단 텍스트를
     공백 정규화(연속 공백·개행→한 칸)한 뒤 **전체 일치** — 부분 일치 아님,
     빈 문단은 못 잡음, 같은 원문이 여럿이면 첫 번째에 붙음. 그래서 앵커는
     read 출력에서 고유한 문단을 그대로 복사할 것. 장을 순서대로 만들면
     "직전 장의 마지막 문단"이 자연스러운 앵커. 새 장 뒤에 표가 삽입된 뒤의
     표 인덱스는 밀리니 table-map으로 재확인. **끼워넣은 장이 기존 번호보다
     앞이면 뒤 장들 번호도 table-set으로 재부여할 것** (예: 예산 5→6)
   - **끼워넣은 장은 새 쪽에서 시작시켜라**: 복제 직후
     `hwpx-kit page-break <파일> --table <새 헤더 표 인덱스> --out <결과>`
     — 안 하면 장 헤더가 페이지 중간에 어중간하게 떨어진다 (실증된 실패).
     문단 기준이면 `--at-text "<문단 원문>"`
   - **앵커 탐색은 `hwpx-kit outline <파일> --json`으로** — 문단 인덱스·
     텍스트·표 위치가 한 번에 나온다. 원시 XML 파싱·자작 스크립트 금지
   - **표 사이에 문단이 없으면** (표와 표가 붙어 있음 — outline으로 확인):
     `--after-text` 대신 `--after-table <표인덱스>` — 그 표 뒤(같은 문단)에
     삽입된다. table-copy·table-new 둘 다 지원
   - 같은 앵커에 여러 개를 넣으면 **넣는 순서대로 아래로 쌓인다** (실측).
     삽입 후 table-map/outline으로 순서 확인하고 어긋나면 조정
   - **임시 앵커 패턴** (그래도 마땅한 앵커가 없을 때): 근처 문단 하나를
     `text:`로 `@@ANCHOR@@` 같은 고유 문구로 바꾸고 → 그걸 앵커로
     table-copy/table-new → 작업 끝나면 `text:@@ANCHOR@@`를 원문으로 되돌리기
     (원래 빈 여백용 문단이었다면 `delete:`로 제거)
   - **승인 절감**: 연속 CLI 단계는 `&&`로 한 셸 호출에 체이닝 (승인 1회).
     사용자가 반복 승인을 귀찮아하면 `/permissions`에서 `hwpx-kit` 또는
     `python -m hwpx_kit.cli` 허용을 안내
2. **내용 확정**: 필드 목록을 사용자에게 보여주고 값을 대화로 확정한다.
   사용자가 원하면 초안을 제안해도 된다. **사용자가 주지 않은 수치·명단
   (예산 금액, 강사명, 정원 등)을 네가 지어냈다면 채우기 전에 반드시
   요약해 보여주고 승인받을 것** — 임의값으로 바로 fill 금지 (실사용 지적)
3. **채우기**: 값을 `{"<fill_key>": "<값>", ...}` 형태의 JSON 파일로 저장 후
   `hwpx-kit fill <양식.hwpx> --data <값.json> --out <결과.hwpx> --json`
   - 원본은 절대 수정되지 않는다. `--out`은 원본과 다른 경로여야 한다
   - 종료 코드 2면 `data.unmatched[]`의 사유를 사용자에게 보여주고 값이나 키를 수정해 재시도
4. **검증**: `hwpx-kit validate <결과.hwpx> --json` — `valid: true` 확인
5. **안내**: 결과 파일 경로를 알려주고, 최종 서식은 한글(한컴오피스)에서 열어 확인하도록 권한다

## 예시 텍스트 교체 (보도자료 등 "예시 문구가 든 양식")

누름틀/빈칸이 아니라 예시 문장이 채워진 양식(보도자료 제목·본문 등)은 `text:` 키 사용:
1. `read --format text`로 원문 확인
2. `{"text:<예시 문장 원문 그대로>": "<새 내용>"}` 으로 fill — 원문과 한 글자라도 다르면 못 찾으니 read 출력에서 복사
3. 서식(폰트/크기/볼드)은 그대로 유지되고 텍스트만 바뀐다 — 표 안이든 본문이든 동작
4. 같은 문구가 문서에 여러 번 있으면 전부 바뀐다 — 고유한 문장 단위로 지정

양식에는 안내문/지시문이 섞여 있다 — "(예시)", "←해당시", "(○○ 관련시)" 표시가 붙은
블록은 실문서에서 지워야 하는 템플릿 지침이다. 해당 없으면 `{"delete:<문단 원문>": ""}`
으로 삭제하고, 해당되면 text:로 실내용으로 교체한다. read 출력에서 이런 블록을 발견하면
사용자에게 삭제/교체 여부를 확인.

강조 요청("이 문장 굵게")은 `{"bold:<원문>": ""}` / `{"underline:<원문>": ""}` —
런 단위 적용이라 원문은 고유한 구 단위로 (자세한 규칙은 gongmun-format 스킬).

## 문서 읽기/요약/검토

`hwpx-kit read <문서> --format md` 출력을 읽고 요약·검토·데이터 추출을 수행한다.
**PDF·DOCX·XLSX도 read로 바로 읽힌다** (내장 파이썬 리더 — 추가 설치 불필요).
구형 .hwp는 ① Windows+한글이면 convert 후 read ② 한글 없는 환경(Mac 등)이면
kordoc 설치 안내 (`npm install -g kordoc`, Node 18+) — read 오류 메시지에 안내 포함.

## 결과 미리보기 / 새 문서 생성

- `hwpx-kit render <문서.hwpx> --out preview.svg --json` — 레이아웃 보존 SVG. fill 후
  브라우저로 열어 서식 확인 가능 (사용자에게 경로 안내)
- `hwpx-kit generate <초안.md> --out <새문서.hwpx> --json` — 양식 없이 새 공문서를 만들 때.
  Markdown으로 작성 후 생성 → validate로 확인

## 워드(docx)로도 달라는 경우

`hwpx-kit export <문서.hwpx> --to docx --out <문서.docx> --json` — 한글 COM으로
같은 내용의 워드 파일을 만든다 (Windows + 한글 필요, convert와 같은 제약).
hwpx를 먼저 완성(fill)한 뒤 export 하는 순서. 결과는 read로 내용 확인 가능.

**변환 직후의 docx는 페이지가 밀려 있다** — 한글과 워드는 조판 엔진이 달라
완벽 일치는 불가능하고, "사람이 마지막 손질" 전제의 초안임을 사용자에게 알린다.
`officecli`가 설치된 환경이면 아래 후처리로 밀림 대부분을 잡을 수 있다
(카이스트 실문서로 검증된 레시피, 2026-07-10):

1. **장 시작 정렬**: 장 제목(번호 달린 헤더 박스, 보통 표)을 찾아 그 직전 빈
   문단에 `pageBreakBefore=true`. 단, 원본에 이미 쪽나눔(run의 breakType=page)이
   있는 문단이면 건너뛴다 — 중복 시 빈 페이지 생김
2. **여백 문단 정리**: 장 사이에 연속된 빈 문단 뭉치(한글에서 빈 줄로 페이지를
   맞춘 흔적)는 워드에서 빈 페이지·표지 밀림의 원인 — 쪽나눔이 대신하므로
   삭제 (쪽나눔 캐리어 1개만 유지). 표지가 넘치면 표지의 빈 문단도 줄인다
3. **음영 찌꺼기 제거**: 한글 변환기가 스타일과 런에 심는 fill 없는 `w:shd`는
   규격상 투명인데 워드가 흰색으로 칠해 색 있는 표 셀 위에 흰 상자가 생긴다.
   `officecli raw-set <파일> /word/styles.xml --xpath '//w:rPr/w:shd[not(@w:fill)]'
   --action remove` — document.xml에도 동일 적용
4. 검증: MS Word가 있으면 COM으로 PDF를 뽑아 페이지별 첫 줄을 확인
   (장 제목이 각 페이지 첫머리인지). 사용자에게 최종 육안 확인 안내

참고: 워드에서 장 시작마다 보이는 작은 검은 네모(■)는 쪽나눔 속성의 편집 기호
표시일 뿐 인쇄되지 않는다 (Ctrl+Shift+8로 숨김) — 사용자가 물으면 이렇게 안내.

## .hwp 파일이 온 경우

먼저 자동 변환 시도: `hwpx-kit convert <문서.hwp> --json` → `data.out`의 hwpx로 이어서 작업.
(Windows + 한글 설치 환경에서만 동작. 한글이 백그라운드로 잠깐 뜰 수 있다고 사용자에게 미리 알린다.)

변환 실패(한글 미설치 등) 시 수동 안내:
1. 한글에서 파일 열기
2. 파일 → 다른 이름으로 저장 → 파일 형식 "HWPX 문서(*.hwpx)" 선택
3. 저장된 .hwpx로 다시 요청

## 보안 — 사용자 제공 양식은 비밀로 취급 (.env급)

부처/기관이 준 양식 파일은 공식 문서다. 절대 금지: git 커밋, 외부 업로드, 공개 경로 복사.
**render로 뽑은 SVG, fill 결과물, read 추출 텍스트 등 파생물도 동일하게 기밀.**
- 양식과 파생물은 `incoming/` 또는 `private/` (둘 다 gitignore) 에만 둔다
- 테스트/스크립트 코드에 양식 속 문장을 하드코딩하지 않는다 (파일에서 동적 추출)
- 공개 출처(정부 공개 서식)만 레포 픽스처로 쓸 수 있다

## 주의

- 암호화된 hwpx는 미지원 — 오류 시 사용자에게 암호 해제 후 재시도 안내
- `warnings`에 미지원 개체 경고가 있으면 그대로 사용자에게 전달 (해당 개체는 보존됨)
- 값에 개인정보가 있을 수 있다 — 채운 결과를 불필요하게 대화에 전문 인용하지 않는다
